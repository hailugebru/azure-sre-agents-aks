Introduction — The Problem Azure SRE Agent Solves

It's 2 AM. Azure Monitor fires a Sev1 alert: pods are crashing in your production AKS cluster. The traditional playbook kicks in — an on-call engineer is paged, logs in, runs a series of diagnostic commands, figures out the root cause, applies a fix, and verifies recovery. By the time that's done, 30 to 60 minutes have passed.

For teams running containerized workloads on Azure Kubernetes Service (AKS), this manual response cycle is a source of alert fatigue, slow recovery times, and inconsistent remediation quality. Azure SRE Agent addresses this head-on: it is an AI-powered operations agent that connects to your Azure resources, receives alerts from Azure Monitor, and executes a structured investigate-diagnose-remediate-verify loop — fully autonomously.

In this post, I'll walk through the complete end-to-end experience: spinning up an AKS cluster with Node Auto-Provisioning (NAP), deploying a realistic demo application, wiring up Azure SRE Agent, and triggering a real OOMKilled incident so you can see the agent resolve it in real time.

Step 1

Deploy an AKS Cluster with Node Auto-Provisioning

The cluster is configured with Node Auto-Provisioning (NAP) — AKS's Karpenter-based node provisioner — combined with Azure CNI Overlay and the Cilium eBPF dataplane. This combination gives you efficient, just-in-time node scaling alongside modern, high-performance networking.

Register the preview feature and create the cluster

Run the prerequisite script to register the NodeAutoProvisioningPreview feature flag (this can take 5–15 minutes), then create the cluster:

# Step 1: Load variables into your session
PS> . .\00-variables.ps1

# Step 2: Register preview features and install aks-preview extension
PS> .\01-prerequisites.ps1

# Step 3: Create the cluster (~5-10 minutes)
PS> .\02-create-cluster.ps1

The cluster creation command under the hood:

az aks create `
  --name "Azure-SRE-Agent-Demo-Cluster" `
  --resource-group "Azure-SRE-Agent-Demo_RG" `
  --node-provisioning-mode Auto            # enables NAP/Karpenter
  --network-plugin azure `
  --network-plugin-mode overlay `
  --network-dataplane cilium               # eBPF — no kube-proxy
  --generate-ssh-keys
💡
Why Cilium? The --network-dataplane cilium flag replaces the legacy iptables/kube-proxy stack with eBPF programs that run directly in the Linux kernel. The result is lower CPU overhead for services with many endpoints, native network policy enforcement, and better observability via Hubble. This is Microsoft's recommended long-term networking configuration for AKS.

Taint the system node pool to activate NAP

By default, NAP only provisions nodes for workloads that can't be scheduled. To make NAP active for the demo app, we taint the system node pool so app pods get evicted and NAP must provision new user nodes:

PS> .\04-setup-nap.ps1

# Watch NAP provision nodes in real time
PS> kubectl get events -A --field-selector source=karpenter -w
PS> kubectl get nodes,pods -n pets -o wide -w

Step 3

Configure Azure SRE Agent

With the cluster and application running, the next step is to wire up the Azure SRE Agent. There are four configuration actions:

  1. Create the Azure SRE Agent resource

    In the Azure portal, search for SRE Agent → + Create. Assign it to the same resource group as your AKS cluster, give it a name (e.g., aks-sre-agent), and enable the system-assigned managed identity.

  2. Grant RBAC to the agent's managed identity

    The agent needs read and patch permissions on the AKS cluster and monitoring resources.

  3. Add your resource group as a Managed Resource

    In the agent's Managed Resources blade, add Azure-SRE-Agent-Demo_RG. This gives the agent visibility over all resources in the group — AKS cluster, Log Analytics workspace, and networking components.

  4. Add an Azure Monitor trigger and configure the Incident Response Plan

    Route Sev0–Sev2 alerts to the agent via an Action Group, then set the agent's operating mode to Autonomous.

Grant RBAC roles

# Replace the --assignee value with your agent's managed identity object ID
$ az role assignment create \
    --assignee "<agent-managed-identity-object-id>" \
    --role "Azure Kubernetes Service Cluster User Role" \
    --scope "/subscriptions/<sub-id>/resourcegroups/Azure-SRE-Agent-Demo_RG"

$ az role assignment create \
    --assignee "<agent-managed-identity-object-id>" \
    --role "Monitoring Reader" \
    --scope "/subscriptions/<sub-id>/resourcegroups/Azure-SRE-Agent-Demo_RG"

Create the pod health alert rule

This rule fires a Sev1 alert whenever the count of failed pods on the cluster is greater than zero for five consecutive minutes:

$ az monitor metrics alert create \
    --name "pod-not-healthy" \
    --resource-group "Azure-SRE-Agent-Demo_RG" \
    --scopes "/subscriptions/<sub-id>/.../managedclusters/Azure-SRE-Agent-Demo-Cluster" \
    --condition "avg kube_pod_status_phase{phase=Failed} > 0" \
    --severity 1 \
    --action "sre-agent-action-group" \
    --evaluation-frequency "1m" \
    --window-size "5m"

Incident Response Plan: three operating modes

ModeBehaviorRecommended Use
Monitor Only Investigates and produces a root cause report. Makes no changes. Initial rollout, compliance-sensitive environments
Guided Proposes exact remediation commands. Waits for human approval before executing. Building team confidence, moderate-severity alerts
Autonomous Full loop: investigate → diagnose → remediate → verify → report. No approval required. Production Sev0/Sev1 where MTTR is the priority
⚡
Recommended ramp-up approach Start in Guided mode for 1–2 weeks. Review and approve the agent's proposed remediations to build operational confidence, then promote to Autonomous for Sev0/Sev1 alerts. The agent never performs destructive operations or escalates its own permissions.

Custom instructions for your environment

The Incident Response Plan supports free-text custom instructions that give the agent domain-specific context about your application. For this demo, we configure:

For AKS pod health alerts in the pets namespace:
1. Scan all namespaces for unhealthy pods first.
2. Prioritise OOMKilled and CrashLoopBackOff.
3. For OOMKilled: correlate NODE_OPTIONS / JVM flags against container memory limits
   before adjusting. Apply the minimum necessary increase.
4. After any patch, wait for rollout, then verify cluster-wide pod health.
5. Create a GitHub issue in the infra-ops repo with the incident summary.
Step 4

See It in Action: Autonomous OOMKilled Recovery

With everything wired up, let's trigger a real incident. We'll change the order-service memory limit to a value that's incompatible with its Node.js V8 heap configuration, causing it to crash repeatedly with OOMKilled.

Trigger the incident

# Set order-service memory limit to 16Mi — incompatible with 64MB V8 heap
PS> kubectl patch deployment order-service -n pets --type='json' `
    -p='[{"op":"replace","path":"/spec/template/spec/containers/0/resources/limits/memory","value":"16Mi"}]'

# Watch pods enter CrashLoopBackOff
PS> kubectl get pods -n pets -w

NAME                               READY   STATUS             RESTARTS   AGE
order-service-5f9d44b6c8-4bvsr     0/1     OOMKilled          1          12s
order-service-5f9d44b6c8-4bvsr     0/1     CrashLoopBackOff   2          28s

Within five minutes, Azure Monitor detects the failing pods and fires the Sev1 alert. The SRE Agent begins working immediately.

The agent's autonomous response — T+0s to T+107s

1
T+0s — Plan

Receive alert, formulate investigation plan

Alert payload received: [Sev1] pod-not-healthy on Azure-SRE-Agent-Demo-Cluster. Agent builds a structured plan: scan all namespaces, prioritise OOMKilled/CrashLoopBackOff, correlate root cause, apply minimal fix, verify recovery.

2
T+5s — Discover

Cluster-wide pod health scan

Connected to the cluster via managed identity. Ran kubectl get pods --all-namespaces and filtered for: OOMKilled, CrashLoopBackOff, ImagePullBackOff, and containers with restarts > 0.

3
T+10s — Identify

Two crashing pods found in pets namespace

Identified order-service-5f9d44b6c8-4bvsr and order-service-5f9d44b6c8-fl782 with 14 restarts each, Last State: OOMKilled.

4
T+20s — Root Cause

Diagnosed: V8 heap incompatible with container memory limit

Correlated four signals: Exit Code 137 (SIGKILL from cgroup OOM killer), Last State: OOMKilled, container memory limit 16Mi, and environment variable NODE_OPTIONS=--max-old-space-size=64. The V8 heap requires 64 MB — 4× the container budget.

5
T+30s — Remediate

Patched deployment: 16Mi → 128Mi memory limit

Applied a minimal JSON patch to the order-service deployment. Triggered a zero-downtime rolling update.

6
T+107s — Verified

Full recovery confirmed, incident report delivered

Both replacement pods Running 1/1, zero restarts. Cluster-wide sweep: zero unhealthy pods. Structured incident report published to the Incident History blade and a GitHub issue created in the infra-ops repo.

Traditional response vs. Azure SRE Agent

Traditional On-Call Response

  • Alert pages on-call engineer at 2 AM
  • Engineer opens laptop, connects to VPN
  • Gets kubeconfig, scans namespaces
  • Reads pod events, looks up exit code 137
  • Decides on fix, applies patch, monitors rollout
  • Writes incident summary the next morning
  • Typical MTTR: 30–60 minutes

Azure SRE Agent (Autonomous Mode)

  • Alert received — agent starts immediately
  • Already authenticated via managed identity
  • Scans all namespaces in parallel
  • Correlates exit codes, specs, and env vars
  • Applies minimal, targeted, reversible patch
  • Verifies rollout and full cluster health
  • MTTR: 1 minute 47 seconds
Bonus

Node Auto-Provisioning in Action

While Azure SRE Agent handles incident response, Node Auto-Provisioning (NAP) handles the infrastructure layer beneath. NAP is AKS's Karpenter integration — instead of you pre-creating node pools with fixed VM sizes, NAP observes pending pods and provisions exactly the right node size and SKU automatically.

Scripts 05-arm-nodepool.ps1 and 06-arm-nodepool-v2.ps1 apply Karpenter NodePool manifests that bias node selection toward ARM64 D-family VMs on Azure Linux (CBL-Mariner) — offering up to 40% better price-performance for Node.js and Go workloads compared to equivalent x86_64 nodes.

PS> .\05-arm-nodepool.ps1   # apply ARM64 NodePool preference
PS> .\06-arm-nodepool-v2.ps1 # add Azure Linux AKSNodeClass

# Inspect NAP provisioned nodes
PS> kubectl get nodepool
PS> kubectl describe nodepool default
