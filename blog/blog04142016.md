# Agentic AI in Production: Autonomous AKS Incident Response with Azure SRE Agent in 8 Minutes

**Tags:** AKS, Azure SRE Agent, AI Ops, Incident Response, Node Auto-Provisioning, KEDA

> **Demo scope:** This walkthrough uses Azure SRE Agent with **Privileged** permissions and **Autonomous** run mode against a dedicated demo resource group. Azure SRE Agent separates **permission levels** (`Reader`, `Privileged`) from **run modes** (`Review`, `Autonomous`), so the safest production rollout is to start narrow and expand deliberately.

---

## The 2 AM problem every SRE team knows

It is 2 AM. A Sev1 alert fires on AKS. In many environments, that still means waking an engineer, correlating logs and metrics, running `kubectl`, applying a fix, and then documenting what happened while half-asleep.

In this demo, Azure SRE Agent did all of that in about 8 minutes without waking anyone. It scanned 40+ pods across six namespaces, isolated the root cause, applied the fix, verified recovery, posted milestone updates to Teams, and created a GitHub issue for follow-up.

That is the value proposition here: not just faster incident response, but a more consistent one. Azure SRE Agent can receive incidents, investigate Azure-native telemetry, reason over infrastructure state, execute mitigations when allowed, verify the outcome, and notify the team through systems such as Microsoft Teams and GitHub.

## Azure SRE Agent on AKS in one minute

Azure SRE Agent is a governed incident-response system, not just a conversational assistant. Five concepts matter most in an AKS incident workflow:

1. **Incident platform** — where incidents originate. In this demo, that is Azure Monitor.
2. **Built-in Azure capabilities** — the agent can use Azure Monitor, Log Analytics, Azure Resource Graph, Azure CLI/ARM, and AKS diagnostics without requiring external connectors.
3. **Connectors** — these extend the workflow to systems such as GitHub, Teams, Kusto, and MCP servers.
4. **Permission levels** — `Reader` for investigation and read-oriented access, `Privileged` for operational changes when allowed.
5. **Run modes** — `Review` for approval-gated execution and `Autonomous` for direct execution.

That model matters more than any single prompt. The most impactful configuration decisions are not what you write in custom instructions. They are which permission level you grant and which run mode you choose.

The safest production rollout path is:

```text
Start:   Reader + Review
Then:    Privileged + Review
Finally: Privileged + Autonomous for narrow, trusted incident paths
```

## Demo environment

The demo combines three layers:

1. an AKS cluster configured for modern scaling and networking patterns,
2. the AKS Store sample application,
3. Azure SRE Agent integrated with Azure Monitor for incident-triggered investigation and remediation.

```text
Azure Monitor  →  Action Group  →  Azure SRE Agent  →  AKS Cluster
(Metric alert)    (Webhook)        (Investigate +       (Diagnose +
                                    Fix + Verify)        Remediate)

                                      ↓
                           Teams notifications + GitHub issue
```

The AKS cluster uses **Node Auto-Provisioning (NAP)**, which is built on Karpenter in AKS, along with Azure CNI Overlay, Cilium, and managed Prometheus metrics. NAP and Azure SRE Agent are complementary but separate: NAP manages infrastructure capacity, while the agent investigates and remediates incidents.

The application layer uses the **AKS Store Demo**: a realistic mix of stateless services, stateful dependencies such as MongoDB and RabbitMQ, and event-driven workers across nine components. KEDA scales `virtual-worker` on RabbitMQ queue depth, adding workload behavior the agent can observe during incident handling.

## Configure Azure SRE Agent

Azure SRE Agent setup in this demo comes down to four decisions: **identity**, **scope**, **incident routing**, and **execution control**.

### 1) Create the agent

Create an Azure SRE Agent resource in the portal and scope it to the demo resource group. During deployment, Azure SRE Agent creates two managed identities: a **user-assigned managed identity (UAMI)** that you use for RBAC and connectors, and a system-assigned identity used internally by the service. The UAMI is the identity that matters for operational setup.

### 2) Grant the right access

Azure SRE Agent uses the UAMI for Azure resource access. Core monitoring roles are assigned during setup, and additional roles depend on your permission level and the resource types in scope. For this demo, I added AKS-specific rights so the agent could perform cluster remediation. Treat those as scenario-specific, not a universal production baseline.

```bash
az role assignment create \
  --assignee "<uami-client-id>" \
  --role "Azure Kubernetes Service Cluster Admin Role" \
  --scope "/subscriptions/<sub-id>/resourcegroups/Azure-SRE-Agent-Demo_RG"
```

### 3) Add the resource group as a managed resource

Add the demo resource group as a managed resource. Managed resource groups define the Azure scope the agent can investigate. Diagnosis quality and available actions still depend on RBAC correctness and telemetry quality.

> _[Screenshot: Azure portal — Managed Resources blade showing `Azure-SRE-Agent-Demo_RG` with status Connected.]_

### 4) Connect Azure Monitor as the incident platform

In this demo, Azure Monitor is the incident platform. It sends the alert payload to Azure SRE Agent through an Action Group webhook, and the agent evaluates the incident against the configured response plan. This is an important distinction: Azure Monitor controls how incidents enter the workflow, while connectors such as GitHub or Teams extend the workflow outward.

> _[Screenshot: Azure SRE Agent portal — Incident Response Plan blade, Autonomous mode selected, allowed actions checklist.]_

### Custom instructions

For this demo, I gave the agent explicit instructions for AKS pod health investigations:

```text
For AKS pod health alerts in the pets namespace:
1. Scan all namespaces for unhealthy pods first.
2. Prioritise OOMKilled and CrashLoopBackOff.
3. For OOMKilled: correlate NODE_OPTIONS / JVM flags against container memory limits before adjusting.
4. After any patch, wait for rollout, then verify cluster-wide pod health.
5. After successful resolution, create a GitHub issue with the incident ID, root cause, patch applied, and a recommendation to update the source manifest in Git.
```

These instructions shape the workflow, but they do not replace RBAC, telemetry quality, or actual tool availability.

Note: Automatic issue creation requires the GitHub connector and a one-time workflow or subagent setup. Once configured, the step runs automatically on every resolved incident.

## Two real incidents, two response modes

These incidents happened on the same cluster in one session and show the two most common ways teams use Azure SRE Agent on AKS:

1. **Alert-triggered automation** — the agent acts when Azure Monitor fires.
2. **Ad hoc chat investigation** — an engineer notices something wrong and hands it off via chat.

### Incident 1 — CPU starvation (alert-driven, ~8 min MTTR)

I intentionally patched `makeline-service` to an unusably low CPU limit:

```powershell
kubectl patch deployment makeline-service -n pets --type='json' `
  -p='[
    {"op":"replace","path":"/spec/template/spec/containers/0/resources/limits/cpu","value":"5m"},
    {"op":"replace","path":"/spec/template/spec/containers/0/resources/limits/memory","value":"20Mi"}
  ]'
```

Within five minutes, Azure Monitor fired the `pod-not-healthy` Sev1 alert (Alert ID `69e4dbba`, fired `15:36:15 UTC`). The agent picked it up immediately.

Here is what the agent’s investigation looked like, using its actual reasoning on exit codes:

> *"Exit code 1 (not 137) rules out OOMKill. The pod failed at startup, not at runtime memory pressure. CPU limit of 5m is insufficient for the process to bind its port before the startup probe times out. This is a configuration error, not a resource exhaustion scenario."*

That single inference, distinguishing a startup probe timeout from an OOMKill, is the kind of domain reasoning that takes an on-call engineer 10 minutes to work through at 2 AM. The agent reached it in under 90 seconds.

The agent then:

- found three more pods CPU-throttled at 112–200% of their limit using `kubectl top`,
- patched four workloads: `makeline-service`, `virtual-customer`, `virtual-worker`, and `mongodb`,
- verified all nine pods returned to Running with 0 restarts cluster-wide.

> _[Screenshot: Azure SRE Agent — Incident History blade showing alert ID `69e4dbba`, Status: Resolved, 4 patches applied, cluster-wide sweep: 0 unhealthy pods.]_

**Outcome:** Full cluster recovery in about 8 minutes. Human interventions: 0.

### Incident 2 — OOMKilled (chat-driven, ~4 min MTTR)

I then deployed a deliberately under-sized version of `order-service`:

```powershell
kubectl apply -f .\manifests\aks-store\order-service-changed.yaml -n pets
```

I started this investigation from chat before the pod-phase alert fired so I could show Azure SRE Agent’s interactive troubleshooting flow. That was a demo choice, not an alerting gap. `CrashLoopBackOff` is a container waiting reason, not a pod phase, so production coverage should come from Prometheus-based crash-loop signals rather than pod phase alone.

```text
The order-service pod in the pets namespace is not healthy.
Please investigate, identify the root cause, and fix it.
```

The agent’s reasoning:

> *"Container logs are empty. The process was killed before it could write its first log line. Exit code 137 confirms OOMKill. No NODE_OPTIONS in the ConfigMap rules out a V8 heap misconfiguration. The 20Mi limit is 12.8× below the pod’s observed 50Mi runtime baseline. This limit was never viable for this workload."*

The agent increased the memory limit (`20Mi → 128Mi`) and request (`10Mi → 50Mi`), then verified the new pod stabilized at 74Mi/128Mi (58% utilization) with 0 restarts.

> _[Screenshot: Azure SRE Agent — chat window showing investigation thread and patch confirmation for `order-service`.]_

**Outcome:** Service recovered in about 4 minutes without any manual cluster interaction.

After remediation, I had the agent create a GitHub issue through the configured GitHub integration. Whether you use the built-in GitHub connector or an MCP-based path, the outcome is the same: the runtime fix becomes a tracked engineering artifact instead of tribal knowledge. Teams gave the on-call team real-time visibility, while GitHub captured the durable follow-up work. Azure SRE Agent notifications are designed to send contextual findings after investigation, not just forward raw alerts.

## Key takeaways

Azure SRE Agent resolved two real AKS incidents in this lab, one alert-driven and one chat-driven, applying 5 targeted patches and filing a complete post-incident GitHub issue, all without waking anyone. The agent also posted milestone updates to the Teams channel so the on-call team could follow the investigation and remediation in real time.

### Three things to carry forward

1. **Azure SRE Agent is a governed incident-response system, not just a chatbot.** The most important controls are permission levels and run modes, not prompt quality.
2. **Built-in Azure diagnostics cover most of the core workflow.** Use connectors to extend the workflow outward, for example Teams for real-time coordination and GitHub for durable engineering follow-up.
3. **Start narrow and expand deliberately.** One resource group, one incident type, `Review` first. Validate that telemetry flows, RBAC is scoped correctly, and your alert rules cover the failure modes you actually care about before enabling `Autonomous`.

### Next steps
