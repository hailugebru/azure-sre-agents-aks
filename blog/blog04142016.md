# Autonomous AKS Incident Response with Azure SRE Agent: From Alert to Verified Recovery in Minutes

**Tags:** Azure SRE Agent, AI Ops, Agentic AI, Incident Response, AKS, KEDA, Azure

---

When a Sev1 alert fires on an AKS cluster, detection is rarely the hard part. The real challenge is fast, consistent response: understanding what broke, why it broke, and fixing it without widening the blast radius. Azure SRE Agent is designed to close that gap by connecting Azure-native observability, AKS diagnostics, and engineering workflows into a single incident-response loop.

This post demonstrates that loop in a production-like AKS lab. In both scenarios, Azure SRE Agent received an incident, investigated Azure Monitor and AKS signals, applied targeted remediation, verified recovery, and created follow-up in GitHub while keeping the team informed in Microsoft Teams. The goal is not to document every edge case, but to show the **power and shape of the new Azure SRE Agent service** when applied to real AKS failures.

## Why this matters

Most AKS environments already emit rich telemetry through Azure Monitor and managed Prometheus. What is still manual is the response: engineers paging through dashboards, running ad-hoc kubectl commands, and applying hotfixes under time pressure. Azure SRE Agent changes that model by turning repeatable investigation and remediation paths into an automated workflow.

In this lab, the impact was clear:


| Metric | Traditional on-call | This demo with Azure SRE Agent |
|---|---|---|
| Alert to recovery | 60–120 min | **~8 min** |
| Human interventions | Multiple | **0** |
| Scope of investigation | Manual, service by service | **Cluster-wide, automated** |
| Correlate evidence and diagnose | 10–15 min | **~2 min** |
| Apply fix and verify | 15–30 min | **~4 min** |
| Post-incident follow-up | Manual | **GitHub issue + draft PR** |

> _These results came from a controlled run on April 10, 2026. Real-world outcomes depend on alert quality, cluster size, and how much automation you enable._

## Azure SRE Agent on AKS in one minute

Azure SRE Agent is a governed incident-response system, not just a conversational assistant. Five concepts matter most in an AKS incident workflow:

1. **Incident platform** — where incidents originate. In this demo, that is Azure Monitor.
2. **Built-in Azure capabilities** — the agent can use Azure Monitor, Log Analytics, Azure Resource Graph, Azure CLI/ARM, and AKS diagnostics without requiring external connectors.
3. **Connectors** — these extend the workflow to systems such as GitHub, Teams, Kusto, and MCP servers.
4. **Permission levels** — `Reader` for investigation and read-oriented access, `Privileged` for operational changes when allowed.
5. **Run modes** — `Review` for approval-gated execution and `Autonomous` for direct execution.

That model matters more than any single prompt. The most important production controls are permission level and run mode. Custom instructions can shape workflow behavior, but they do not replace RBAC, telemetry quality, or tool availability.

The safest production rollout path is:

```text
Start:   Reader + Review
Then:    Privileged + Review
Finally: Privileged + Autonomous for narrow, trusted incident paths
```

## How Azure SRE Agent was configured for this demo

For this demo, Azure SRE Agent configuration came down to four things: scope, permissions, incident intake, and response mode. I scoped the agent to the demo resource group and used its user-assigned managed identity (UAMI) for Azure access. That scope defined what the agent could investigate, while RBAC determined what actions it could take. 

I used broader AKS permissions than I would recommend as a default production baseline so the agent could complete remediation end to end in the lab. That is an important distinction: permissions control what the agent can access, while run mode controls whether it asks for approval or acts directly. For this scenario, Azure Monitor served as the incident platform, and I set the response plan to Autonomous for a narrow, trusted path so the workflow could run without manual approval gates.

I also added Teams and GitHub integrations so the workflow could extend beyond Azure itself: Teams for milestone updates during the incident and GitHub for durable follow-up after remediation. The agent’s built-in Azure capabilities handled the core investigation; the integrations mainly extended communication and tracking. Custom instructions helped shape the AKS pod-health workflow, but the real production controls were still scope, RBAC, telemetry quality, and run mode. For the complete Azure SRE Agent setup, see the README.

## Two common AKS incident patterns, two response modes

These incidents occurred on the same cluster in one session and illustrate two realistic operating modes when teams use Azure SRE Agent on AKS:

1. **Alert-triggered automation** — the agent acts when Azure Monitor fires.
2. **Ad hoc chat investigation** — an engineer sees a symptom first and asks the agent to investigate. 

Both matter in real environments. The first is your scale path. The second is your operator-assist path.

### Incident 1 — CPU starvation (alert-driven, ~8 min MTTR)

In the first case, makeline-service was already defined in the deployment manifest with a CPU and memory configuration that was not viable for startup:

```yaml
resources:
  requests:
    cpu: 1m
    memory: 6Mi
  limits:
    cpu: 5m
    memory: 20Mi
```

Within five minutes, Azure Monitor fired the `pod-not-healthy` Sev1 alert. The agent picked it up immediately.

Here is the key diagnostic conclusion the agent reached from the pod state, probe behavior, and exit code:

> *"Exit code 1 (not 137) rules out OOMKill. The pod failed at startup, not at runtime memory pressure. CPU limit of 5m is insufficient for the process to bind its port before the startup probe times out. This is a configuration error, not a resource exhaustion scenario."*

That is the kind of distinction that often takes an on-call engineer several minutes to prove under pressure: startup failure from CPU starvation vs. runtime termination from memory pressure.

The agent then:

- identified three additional CPU-throttled pods at 112–200% of configured limit using `kubectl top`,
- patched four workloads: `makeline-service`, `virtual-customer`, `virtual-worker`, and `mongodb`,
- verified that all affected pods returned to healthy running state with 0 restarts cluster-wide.

> _[Screenshot: Azure SRE Agent — Incident History blade showing alert ID `69e4dbba`, Status: Resolved, 4 patches applied, cluster-wide sweep: 0 unhealthy pods.]_

**Outcome:** full cluster recovery in about 8 minutes, with 0 human interventions.

### Incident 2 — OOMKilled (chat-driven, ~4 min MTTR)

For the second case, I deployed a deliberately under-sized version of `order-service`:

```powershell
kubectl apply -f .\manifests\aks-store\order-service-changed.yaml -n pets
```

I intentionally started the second case from chat before the pod-phase alert fired to demonstrate the interactive troubleshooting flow. That was a demo choice, not an alerting gap. `CrashLoopBackOff` is a container waiting reason, not a pod phase, so production coverage should come from Prometheus-based crash-loop signals rather than pod phase alone.

```text
The order-service pod in the pets namespace is not healthy.
Please investigate, identify the root cause, and fix it.
```

The agent’s reasoning:

> *"Container logs are empty. The process was killed before it could write its first log line. Exit code 137 confirms OOMKill. No NODE_OPTIONS in the ConfigMap rules out a V8 heap misconfiguration. The 20Mi limit is 12.8× below the pod’s observed 50Mi runtime baseline. This limit was never viable for this workload."*

The agent increased the memory limit (`20Mi → 128Mi`) and request (`10Mi → 50Mi`), then verified the new pod stabilized at 74Mi/128Mi (58% utilization) with 0 restarts.

> _[Screenshot: Azure SRE Agent — chat window showing investigation thread and patch confirmation for `order-service`.]_

**Outcome:** Service recovered in about 4 minutes without any manual cluster interaction.

## Teams + GitHub follow-up

Runtime remediation is only half the story. If the workflow ends when the pod becomes healthy again, the same issue will likely return on the next deployment.
That is why the post-incident path matters.

After Incident 1 resolved, Azure SRE Agent created a GitHub issue with the remediation details. I then assigned that issue to GitHub Copilot agent, which analyzed the repository and opened a draft pull request to align the source manifests with the incident hotfix for my review. This extended the workflow beyond runtime remediation: Azure SRE Agent fixed the live issue, and GitHub Copilot agent prepared the durable source change so future deployments would not reintroduce the same configuration problem. Human review still remained the final control point before merge.

> _[Screenshot: Draft pull request created by GitHub Copilot agent showing order-service memory request/limit updates to match the incident hotfix.]_

In parallel, I added a Microsoft Teams connector so the agent could post milestone updates during the incident:

- investigation started
- root cause and remediation identified
- incident resolved

Teams handled real-time situational awareness. GitHub handled durable engineering follow-up. Together, they closed the gap between operations and software delivery. For GitHub integration details, see the README.

## Key takeaways

In this AKS lab, Azure SRE Agent handled two common incident patterns—one alert-driven and one chat-driven—applied targeted remediation, verified recovery, posted Teams updates, and created GitHub follow-up. The workflow then continued into source control, where the generated issue was assigned to GitHub Copilot agent to produce a draft pull request for review.

### Three things to carry forward

1. **Treat Azure SRE Agent as a governed incident-response system, not a chatbot with infrastructure access.** The most important controls are permission levels and run modes, not prompt quality.
2. **Anchor detection in Prometheus and Azure.** Use connectors to extend the workflow outward, for example Teams for real-time coordination and GitHub for durable engineering follow-up.
3. **Start narrow and expand deliberately.** One resource group, one incident type, `Review` first. Validate that telemetry flows, RBAC is scoped correctly, and your alert rules cover the failure modes you actually care about before enabling `Autonomous`.

### Next steps

- Add Prometheus-based alert coverage for `ImagePullBackOff` and node resource pressure to complement the pod phase rule.
