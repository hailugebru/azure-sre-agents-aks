# Agentic AI in Production: Autonomous AKS Incident Response with Azure SRE Agent in 8 Minutes

**Tags:** AKS, Azure SRE Agent, AI Ops, Incident Response, Node Auto-Provisioning, KEDA

> **Demo scope:** This walkthrough uses Azure SRE Agent with **Privileged** permissions and **Autonomous** run mode against a dedicated demo resource group. Azure SRE Agent separates **permission levels** (`Reader`, `Privileged`) from **run modes** (`Review`, `Autonomous`), so the safest production rollout is to start narrow and expand deliberately.

## Introduction

It’s 2 AM. A Sev1 alert fires on AKS. In many environments, that still means waking an engineer, correlating logs and metrics, running `kubectl`, applying a fix, and then documenting what happened. That process is slow, inconsistent, and expensive in human attention.

By the time a human on-call engineer would have connected to the cluster, Azure SRE Agent had already scanned 40+ pods across six namespaces, isolated the root cause, and was applying the first patch. Total elapsed time from alert to full cluster recovery: **8 minutes, zero human interventions**.

Azure SRE Agent is designed to reduce that toil. It can receive incidents, investigate Azure-native telemetry, reason over infrastructure state, propose or execute mitigations, verify the outcome, and capture the result as an operational artifact. In this demo, Azure Monitor acts as the **incident platform**, Azure diagnostics come from the agent’s built-in Azure capabilities, and GitHub is added as an external system through a connector.

This walkthrough shows that end-to-end flow on Azure Kubernetes Service (AKS): a lightweight environment setup, Azure SRE Agent configuration, and two real Kubernetes failure modes resolved without human intervention.

## Azure SRE Agent on AKS in one minute

Azure SRE Agent has five concepts that matter most in an AKS incident workflow:

- **Incident platform** — where incidents originate. In this demo, that is Azure Monitor.
- **Built-in Azure capabilities** — the agent can use Azure Monitor, Log Analytics, Azure Resource Graph, Azure CLI/ARM, and AKS diagnostics without requiring external connectors.
- **Connectors** — these extend the agent to external systems such as GitHub, Teams, Kusto, and MCP servers.
- **Permission levels** — `Reader` for investigation and read-oriented access, `Privileged` for operational changes when allowed.
- **Run modes** — `Review` for approval-gated execution and `Autonomous` for direct execution.

That model matters more than any single prompt. **Azure SRE Agent is a governed incident-response system, not just a conversational assistant.** The most impactful configuration decisions are not what you write in the custom instructions — they are which permission level you grant and which run mode you choose.

## Demo environment

The demo combines three layers:

1. an AKS cluster configured for modern scaling and networking patterns,
2. the AKS Store sample application,
3. Azure SRE Agent integrated with Azure Monitor for incident-triggered investigation and remediation.

```text
Azure Monitor  →  Action Group  →  Azure SRE Agent  →  AKS Cluster
(Metric alert)    (Webhook)        (Investigate +       (NAP + Cilium
                                    Fix + Verify)         + KEDA)
```

The AKS cluster uses **Node Auto-Provisioning (NAP)**, which is built on Karpenter in AKS, along with Azure CNI Overlay, Cilium, and managed Prometheus metrics. NAP and Azure SRE Agent are complementary but separate: NAP manages infrastructure capacity, while the agent investigates and remediates incidents.

The application layer uses the **AKS Store Demo** — a realistic mix of stateless services, stateful dependencies (MongoDB, RabbitMQ), and event-driven workers across nine components. KEDA scales `virtual-worker` on RabbitMQ queue depth, adding workload behavior the agent can observe during incident handling.

> **Alert coverage gotcha:** `kube_pod_status_phase{phase="Failed"}` is useful, but it does not cover every Kubernetes failure mode. `CrashLoopBackOff` is a **container waiting reason**, not a pod phase — it won't trigger this rule alone. Production-grade alerting should combine pod phase, waiting reason, termination reason, and node health signals. Incident 2 below would have gone undetected by the alert rule.

## Configure Azure SRE Agent

Azure SRE Agent setup in this demo comes down to four decisions: **identity**, **scope**, **incident routing**, and **execution control**.

### 1) Create the agent

Create an Azure SRE Agent resource in the portal and scope it to the demo resource group. During deployment, Azure SRE Agent creates two managed identities: a **user-assigned managed identity (UAMI)** that you use for RBAC and connector scenarios, and a system-assigned identity used internally by the service.

> _[Screenshot: Azure portal — Create Azure SRE Agent, Basics tab showing subscription, resource group, agent name, region, and managed identity toggle set to On.]_


### 2) Grant the right access

Azure SRE Agent uses the UAMI for Azure resource access. Core monitoring roles are assigned during setup — including Reader, Log Analytics Reader, Monitoring Reader, and **Monitoring Contributor**.

> **Gotcha:** Monitoring Contributor is assigned at **subscription scope**, not just the resource group scope. This surprised several teams during preview. Review it against your organization's change-control and least-privilege requirements before production rollout.

Additional roles depend on your permission level and the resource types in your managed scope. In this demo, I added AKS-specific rights for cluster remediation. Treat these as scenario-specific, not a universal production baseline:

```bash
az role assignment create \
  --assignee "<uami-client-id>" \
  --role "Azure Kubernetes Service Cluster Admin Role" \
  --scope "/subscriptions/<sub-id>/resourcegroups/Azure-SRE-Agent-Demo_RG"

az role assignment create \
  --assignee "<uami-client-id>" \
  --role "Azure Kubernetes Service Contributor Role" \
  --scope "/subscriptions/<sub-id>/resourcegroups/Azure-SRE-Agent-Demo_RG"
```

### 3) Add the resource group as a managed resource

Managed resource groups define the Azure scope the agent can investigate. Diagnosis quality and available actions still depend on RBAC and telemetry quality.

> _[Screenshot: Azure portal — Managed Resources blade showing `Azure-SRE-Agent-Demo_RG` with status Connected.]_

### 4) Connect Azure Monitor as the incident platform

Configure Azure Monitor in the Azure SRE Agent portal, copy the generated webhook URL, and route your AKS alert to it through an Azure Monitor Action Group. When the alert fires, Azure Monitor sends the incident payload to the agent, which evaluates it against the response plan and starts investigating. Azure Monitor is the **incident platform** in this design — not just another connector.

> _[Screenshot: Azure SRE Agent portal — Builder, Incident platform, Azure Monitor connected with webhook URL and active response plan count.]_

### Operating model: permissions and run modes

For production adoption, think in two layers:

- **Permission level** — what resources the agent can access and modify
- **Run mode** — whether the agent asks before acting

| Pattern | Permission | Run mode | Behavior | Good for |
|---|---|---|---|---|
| Investigation only | `Reader` | `Review` | Investigates and reports. No changes. | First rollout, compliance environments |
| Guided remediation | `Privileged` | `Review` | Proposes actions, waits for human approval. | Building confidence, moderate-severity alerts |
| Autonomous remediation | `Privileged` | `Autonomous` | Full loop: investigate, fix, verify, report. | Trusted, narrow Sev0/Sev1 paths |

A pragmatic rollout: start with `Reader` + `Review` for 1–2 weeks, promote to `Privileged` + `Review`, then `Autonomous` for your highest-confidence incident types.

> _[Screenshot: Azure SRE Agent portal — Incident Response Plan blade, Autonomous mode selected, allowed actions checklist.]_

### Custom instructions

For this demo, I gave the agent explicit instructions for AKS pod health investigations:

```text
For AKS pod health alerts in the pets namespace:
1. Scan all namespaces for unhealthy pods first.
2. Prioritise OOMKilled and CrashLoopBackOff.
3. For OOMKilled: correlate NODE_OPTIONS / JVM flags against container memory limits before adjusting.
4. After any patch, wait for rollout, then verify cluster-wide pod health.
5. After successful resolution, create a GitHub issue with the incident ID, root cause, patch applied, and a recommendation to update the source manifest in Git.
```

These instructions shape the workflow, but they do not replace RBAC, telemetry quality, or actual tool availability.

> **Gotcha:** Instruction 5 (create a GitHub issue) requires a GitHub connector **and** a dedicated subagent configured in the Subagent Builder. Without that wiring, the agent will attempt the step and fail silently. The connector and subagent setup is a one-time configuration — once done, it fires automatically on every resolved incident.

### Post-incident GitHub issue

After remediation, I had the agent create a GitHub issue via the GitHub MCP connector. Azure SRE Agent supports GitHub integration through its built-in connector; MCP-based integration gives more explicit tool control. Either way, the outcome is the same: the runtime fix becomes a tracked engineering artifact instead of tribal knowledge.

After Incident 1 resolved, the agent automatically filed [Issue #1](https://github.com/hailugebru/azure-sre-agents-aks/issues/1) at `16:01:26 UTC` — no human action. The issue included:

- Alert ID `69e4dbba`, timestamp, and namespace scope
- All unhealthy pods with failure states and root cause per failure type
- Before/after resource values for all 5 patches
- Post-patch `kubectl top` verification showing all 9 pods healthy
- 7 actionable recommendations, including: *"Update source manifests in Git"* and *"Add resource limit validation to the CI/CD pipeline to prevent sub-10m CPU limits from being deployed"*
- A deep link to the SRE Agent investigation thread for a full audit trail

> _[Screenshot: GitHub Issue #1 — title `[AKS Alert] Pod Health Failures - pets namespace - 69e4dbba`, labels (aks, memory-management, pod-health), and recommendation body.]_

## Two real incidents

These incidents happened on the same cluster in one session and show the two most common ways teams use Azure SRE Agent on AKS:

1. **alert-triggered automation** — the agent acts when Azure Monitor fires
2. **ad hoc chat investigation** — an engineer noticed something wrong and handed it off via chat

### Incident 1 — CPU starvation (alert-driven, ~8 min MTTR)

I intentionally patched `makeline-service` to an unusably low CPU limit:

```powershell
kubectl patch deployment makeline-service -n pets --type='json' `
  -p='[
    {"op":"replace","path":"/spec/template/spec/containers/0/resources/limits/cpu","value":"5m"},
    {"op":"replace","path":"/spec/template/spec/containers/0/resources/limits/memory","value":"20Mi"}
  ]'
```

Within five minutes, Azure Monitor fired the `pod-not-healthy` Sev1 alert (Alert ID `69e4dbba`, fired `15:36:15 UTC`). The agent picked it up immediately.

Here is what the agent's investigation looked like — its actual reasoning on exit codes:

> *"Exit code 1 (not 137) rules out OOMKill. The pod failed at startup, not at runtime memory pressure. CPU limit of 5m is insufficient for the process to bind its port before the startup probe times out — this is a configuration error, not a resource exhaustion scenario."*

That single inference — distinguishing a startup probe timeout from an OOMKill — is the kind of domain reasoning that takes an on-call engineer 10 minutes to work through at 2 AM. The agent reached it in under 90 seconds.

The agent then:

- found three more pods CPU-throttled at 112–200% of their limit using `kubectl top`,
- patched four workloads (`makeline-service`, `virtual-customer`, `virtual-worker`, `mongodb`),
- verified all nine pods returned to Running/0 restarts cluster-wide.

> _[Screenshot: Azure SRE Agent — Incident History blade showing alert ID `69e4dbba`, Status: Resolved, 4 patches applied, cluster-wide sweep: 0 unhealthy pods.]_

**Outcome:** full cluster recovery in ~8 minutes. Human interventions: 0.

### Incident 2 — OOMKilled (chat-driven, ~4 min MTTR)

I then deployed a deliberately under-sized version of `order-service`:

```powershell
kubectl apply -f .\manifests\aks-store\order-service-changed.yaml -n pets
```

No alert fired — this was `CrashLoopBackOff`, not a pod phase failure. An engineer noticed the pod unhealthy and handed it off via chat:

```text
The order-service pod in the pets namespace is not healthy.
Please investigate, identify the root cause, and fix it.
```

The agent's reasoning:

> *"Container logs are empty — the process was killed before it could write its first log line. Exit code 137 confirms OOMKill. No NODE_OPTIONS in the ConfigMap rules out a V8 heap misconfiguration. The 20Mi limit is 12.8× below the pod's observed 50Mi runtime baseline — this limit was never viable for this workload."*

The agent increased the memory limit (`20Mi → 128Mi`) and request (`10Mi → 50Mi`), then verified the new pod stabilized at 74Mi/128Mi (58% utilization) with 0 restarts.

> _[Screenshot: Azure SRE Agent — chat window showing investigation thread and patch confirmation for order-service.]_

**Outcome:** service recovered in ~4 minutes without any manual cluster interaction.

> **Note:** I intentionally started this investigation from chat before Azure Monitor fired so I could show Azure SRE Agent’s interactive troubleshooting flow. This was a demo choice, not an alerting gap. The same crash-loop condition was later detected automatically through Prometheus-based alerting, and Azure SRE Agent created **GitHub Issue #4** through the incident-driven workflow. In production, that is the model I recommend: automatic detection through Azure Monitor / Prometheus, with chat available when an engineer wants to investigate interactively. `CrashLoopBackOff` is a container waiting reason, not a pod phase.

## Key takeaways

Azure SRE Agent resolved two real AKS incidents in this lab — one alert-driven, one chat-driven — applying 5 targeted patches and filing a complete post-incident GitHub issue, all without waking anyone.

The key value is not just speed. It is **consistency**: the same investigate → diagnose → remediate → verify → report loop every time, whether it's 2 AM or 2 PM.

### Three things to carry forward

1. **Azure SRE Agent is a governed incident-response system, not just a chatbot.** The most important controls are **permission levels** and **run modes** — not prompt quality.
2. **Built-in Azure diagnostics cover most of the core workflow.** Use connectors when you want the agent to extend into GitHub, Teams, or other external systems.
3. **Start narrow and expand deliberately.** One resource group, one incident type, `Review` first. Validate that telemetry flows, RBAC is scoped correctly, and your alert rules cover the failure modes you actually care about before enabling `Autonomous`.

### Next steps

- Add alert coverage for `CrashLoopBackOff` (container waiting reason), `ImagePullBackOff`, and node resource pressure to complement the pod phase rule
- Add Teams notifications for post-incident handoff alongside the GitHub issue
- Expand to multi-cluster managed scopes once the single-cluster path is trusted and validated
